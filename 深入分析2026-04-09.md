一、Auto Dream 机制升级方案

1.1 现状诊断

当前 services/autoDream/autoDream.ts 的实现本质是**"定时批量反思"**：
- 触发 = hoursSince ≥ 24h && sessionsSince ≥ 5 && lock 可拿
- 执行 = 一次性 runForkedAgent，把 /dream prompt 跑一遍，产出写入 ~/.claude/memory/*
- 状态 = 只有一把 consolidationLock + lastConsolidatedAt 单一标量

五个底层问题：

├─────┼────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ P1  │ "全量重读"浪费 │ 每次 dream 都拿全部 session 列表，LLM 自己取样，上下文极易打爆，cache_read 增长不受控               │

├─────┼────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────┤

└─────┴────────────────┴─────────────────────────────────────────────────────────────────────────────────────────────────────┘

1.2 方法论重构：从"批 compact"到"睡眠周期模型"

借鉴人类睡眠的 NREM/REM 分期，把 dream 拆成五阶段流水线，每一阶段都是独立、可中断、可重入的 side-query：

Session Journal (写)
│
├── Stage-1 EPISODIC INDEX   （每 session 结束，P3_background）
│      抽取：user_intent / tools_used / files_touched / outcome
│      产物：~/.claude/memory/episodes/{sid}.jsonl（结构化，非自然语言）
│
├── Stage-2 SEMANTIC CLUSTER （N 个 episode 累计后触发，P3）
│      对 episode embedding 做在线聚簇（MinHash+LSH 或 IVF 即可）
│      产物：cluster_id → 代表 episodes 列表 + 主题关键词
│
├── Stage-3 CONSOLIDATION    （按 cluster 触发，而非按时间）
│      只把同一 cluster 的 episodes 送进 forked agent
│      每次 prompt 长度上限 ≤ 8k，cache 命中率高
│      产物：memory 文件（按 cluster 命名，避免同一主题反复被重写）
│
├── Stage-4 VERIFICATION     （consolidation 后立即跑，P2_method）
│      用小模型/规则校验：新 memory 是否和老 memory 冲突？
│      冲突 → 生成 patch 建议或标注 deprecated；一致 → 合并
│
└── Stage-5 FORGETTING       （每周跑一次，P3）
对 memory 打分：recall_hits / age / last_hit_ts
低分的进 archive/，不删除、只降权（可逆）

1.3 关键设计点

(a) Episodic Index 不走 LLM
Session 结束时的抽取用规则管线（正则 + tool_use 统计），零 token 成本、零网络。只把结构化 JSONL 落盘。这是把"dream 的耗能"从 O(sessions) 降到
O(clusters) 的前提。

(b) 调度由 SideQueryScheduler 统一接管
当前 autoDream 自己维护 lastSessionScanAt 节流器。应改成：
- 注册为 category: 'auto_dream', priority: 'P3_background'
- 触发源 = 事件驱动 + 保底定时，不是纯定时
    - 事件：episode_buffer > N / cluster_dirty / idle_for > 10min
    - 保底：24h 没触发则强制一次轻量 Stage-5
- 预算、熔断、去重全部复用 sideQuery/scheduler.ts，不需要 autoDream 自己实现节流

(c) Forking 参数按阶段定制
Stage-3 用 forked agent（目前这条路径）；Stage-1/2/4/5 全部不走 LLM或走小模型。这意味着 forkedAgent 只承担"真正需要生成"的部分，cache_creation
降一个数量级。

(d) Lock 结构化
把 consolidationLock 从"单一 mtime"升级为：
{
"stage": "stage3_consolidation",
"cluster_id": "auth-middleware",
"attempt": 2,
"last_error": "api_timeout",
"started_at": "..."
}
失败时只回滚当前 stage+cluster，其它 cluster 的产物保留。rollbackConsolidationLock 变成"restage"而不是"rewind"。

(e) 召回反馈闭环
新增 memory_recall 命中埋点（每次 getRelevantMemoryAttachments 返回非空都记一笔 memoryId → hit++）。Stage-5 用这个 counter 做 forgetting
分数，形成**"被用到的记忆更不易被淡忘"**的正反馈。这是目前彻底缺失的环。

(f) 冲突检测走规则而非 LLM
Stage-4 用 diff 工具对比新老 memory 文件：
- 同一 frontmatter name 字段但 body 相似度 > 80% → 合并
- 相似度 20-80% → 标注 conflicted: true，下一次 Stage-3 带着冲突上下文重跑
- < 20% → 视为新记忆

这样避免"两份记忆悄悄互相覆盖"的静默错误。

2.3 关键升级点

(a) L0 Intent Classifier（免费且必须）
新建 skillSearch/intent.ts，用 50-100 条正则/关键词规则把 user input 映射到 TaskMode 枚举（code_edit / debug / shell_ops / git_workflow /
data_query / docs_read ...）。零 token 成本。TaskMode 作为召回的第一过滤器和L2 rerank 的 feature。

(b) L1 Embedding 通路
- 用本地 tiny embedding（all-MiniLM-L6 int8 / bge-small），skill 索引构建时离线跑一遍，存 ~/.claude/skill-index.bin
- Query-time 只对 user message 做一次 embed + HNSW top-k
- 和现有 keyword RRF 并行而非替代——embedding 负责召回泛化，keyword 负责精确命中。RRF 自动兜住两者
- 失败/模型缺失 → 降级到纯 keyword，零回归

(c) L2 Learning-to-Rank（轻量线性模型即可）
特征向量：
[keyword_rrf, embed_rrf, intent_match, hit_rate_30d,
last_used_ago, same_author_recent, risk_score, token_cost]
权重靠离线训练/或手动初始 + 在线纠偏（Bandit）。无需深度模型，一行加权求和就够了。重点是有这个 feature 层，未来任何新 signal 都能接进来。

(d) 多触发点 + 增量召回
当前只在 user message 入口召回。应扩展到：
- pre_tool_use：assistant 即将调用 Bash/Edit 时，对 tool_use.input 做一次 mini 召回（category='skill_discovery_midloop', priority='P2_method'）
- post_error：工具失败时，用 error message 做一次召回（debug 类 skill 专属）
- compact_boundary：compact 后重新召回一次，恢复被压缩掉的 skill 上下文

所有入口都走 submitSideQuery + dedupe，不会重复跑。

(e) 反馈闭环
- 每召回一个 skill，记录 recall_id → 塞进 attachment metadata
- 后续若看到 tool_use: SkillTool(name=X) 或 assistant 明确提及，则 hit=true
- 若整个 turn 结束都没命中，则 hit=false + 降 taskMode 下该 skill 权重
- 数据落 ~/.claude/skill-stats.json，按 skill×taskMode 二维矩阵
- 这份矩阵是 L2 rerank 的燃料，也是 dream Stage-5 forgetting 的输入

(f) 协同式去重
现在 discoveredSkillNames 是全局一次性 Set。改成：
- Session 级 LRU（默认 20 条），按使用时间淘汰
- 按 taskMode 分桶，切换 mode 时自动清掉上一 mode 的抑制
- 若某 skill 被召回 N 次都没命中，进入 "cooldown" 状态 15 分钟

2.4 部署与风险

- 全部新代码默认 OFF（CLAUDE_SKILL_RECALL_V2=1），shadow 跑 1 周对比命中率
- L0/L2 可独立开启，互不依赖
- L1 embedding 若模型未安装，自动降级成当前双维 RRF

2.5 举一反三

同一"多阶段漏斗 + 特征层 rerank + 反馈闭环"直接迁移到：
- Memory recall（getRelevantMemoryAttachments）：目前也是简单词频，同样需要 L0 intent + L1 embedding + L2 rerank
- MCP tool 选择：MCP server 暴露的 tool 列表越来越长，同样需要按 intent 过滤
- Agent tool 分派（subagent_type 选择）：目前靠 prompt 文字匹配，可以走完全一样的漏斗

核心抽象：应在 services/retrieval/ 下沉一套 Retriever 接口（query → scored items）和 Reranker 接口（items + features → ordered
items），skill/memory/mcp-tool/agent 四个场景共用一套，只换数据源和 feature 提取器。这就是 harness 层真正该提供的召回基础设施。

3.2 方法论重构：PEV Loop (Plan → Execute → Verify) + Effect Ledger

把"自动运行代码"从单条 tool call 语义升级为一个完整的执行平面：

┌──────────────────────────────────────────────────────────┐
│                     Execution Plane                      │
│                                                          │
│  ┌─────────┐   ┌───────────┐   ┌─────────┐  ┌─────────┐  │
│  │ Planner │──►│ Policy    │──►│Executor │─►│Verifier │  │
│  │ (LLM)   │   │ Engine    │   │         │  │         │  │
│  └─────────┘   └─────┬─────┘   └────┬────┘  └────┬────┘  │
│                      │              │            │       │
│                      ▼              ▼            ▼       │
│                Effect        Snapshot        Effect      │
│                Predictor     Manager         Ledger      │
│                      │              │            │       │
│                      └──────────────┴────────────┘       │
│                         Audit/Rollback/Replay           │
└──────────────────────────────────────────────────────────┘

3.3 关键组件

(a) Planner（LLM 阶段）
把当前 "LLM 直接调 Bash 一条命令" 改成 "LLM 先提交 ExecutionPlan"：
{
"steps": [
{"op": "bash", "cmd": "pnpm install X",
"predicted_effects": ["write:node_modules", "write:package.json"],
"verify": {"type": "file_exists", "path": "node_modules/X"},
"rollback": {"op": "bash", "cmd": "pnpm remove X"}
}
],
"goal": "add X dependency"
}
这个 plan 先进 Policy Engine 审批，审批通过后 Executor 一次性执行并验证。单条命令场景可以自动合成 plan（兼容现有使用），不强制 LLM 显式生成。

(b) Policy Engine（基于 effect 的权限）
权限规则从 "工具 + 正则" 升级为 "effect 类型 + 资源路径"：
# ~/.claude/exec-policy.yml
allow:
- effect: read          # 任何读
- effect: write
path_glob: "src/**"
- effect: network
host: "registry.npmjs.org"
deny:
- effect: write
path_glob: ".github/workflows/**"
- effect: exec
cmd_glob: "rm -rf *"
auto_approve_if:
- effect: write
path_glob: "src/**"
branch_is_not: ["main", "master"]
当前的 bashSecurity/bashPermissions 成为此 engine 的一个 backend（命令 → effect 翻译层）。

(c) Effect Predictor
一个纯规则模块：输入命令/文件编辑，输出预期 effect 集合。例如：
- pnpm install → {write:node_modules, write:pnpm-lock.yaml, network:registry}
- git push → {network:git-remote, external-visible}
- Edit file=X → {write:X}
- rm -rf → {destructive-write, irreversible}

所有"危险 vs 安全"判断 都基于 effect 标签，不再基于命令串模式匹配。正则易被绕过，effect 集合可枚举可审计。

(d) Snapshot Manager
在 Executor 执行前，对 plan 涉及的文件/目录做廉价快照：
- 文件级：内容 hash + 内容本身（小文件）或 cp 到 .claude/snapshots/<plan_id>/
- git 工作区级：git stash create 产出一个 stash hash（不入栈，只留引用）
- 进程级：不 snapshot，只记录 pid/port，用于事后 cleanup

成功 → 30 分钟后 GC；失败 → 自动 rollback。这把"git 兜底"从外部补丁变成平台能力。

(e) Verifier
每 step 定义 verify 字段（plan 由 LLM 填，或 Planner 规则自动合成）：
- exit_code == 0
- file_exists
- regex_in_output
- tool_specific（如 pnpm test 要求 0 failed）

失败则 → Planner 阶段决定 retry / rollback / ask_user，而不是直接把错误抛回 LLM 主循环。

(f) Effect Ledger
所有成功的 step 把 effects 追加进 ~/.claude/exec-ledger.jsonl：
{"ts":"...","plan_id":"...","step":0,"effects":[...],"snapshot":"...","rollback":"..."}
作用：
1. /undo 命令：读最后 N 条 entry，按 rollback 字段反向执行
2. 相干检查：下一个 plan 的 effect 和已有 ledger 冲突（如前一步 rm file、后一步 read file）直接 Policy 拒绝
3. 审计：一天跑了什么，一键导出

(g) Sandbox 多后端
- macOS：保留现有 sandbox-exec
- Linux：landlock（>=5.13）+ seccomp 白名单
- 容器：Docker/podman 的 --read-only + bind mount
- 通用 fallback：cwd chroot + PATH 剔除（能干的事少，安全性最低但总比没有强）

Policy Engine 按平台自动选择，LLM 不感知。

(h) 失败自愈策略表
不再把原始 error 扔给 LLM 让它自己想。Verifier 失败后先查策略表：

┌─────────────────────────────┬─────────────────────────────┐
│        Error pattern        │           Action            │
├─────────────────────────────┼─────────────────────────────┤
│ ENOENT + file in edit input │ retry once after Glob       │
├─────────────────────────────┼─────────────────────────────┤
│ command not found           │ suggest install, ask user   │
├─────────────────────────────┼─────────────────────────────┤
│ permission denied           │ elevate path policy request │
├─────────────────────────────┼─────────────────────────────┤
│ network timeout             │ retry with backoff (≤2)     │
├─────────────────────────────┼─────────────────────────────┤
│ `git merge                  │                             │
└─────────────────────────────┴─────────────────────────────┘

一、Auto Dream 机制升级：从"定时巩固"→"证据驱动的记忆生命周期引擎"

现状问题（源自 services/autoDream/autoDream.ts）

1. 三段硬门控：minHours=24 + minSessions=5 + 文件锁。判据粗粒度，只数 session 数量、不看内容质量。一个用户连开 5 个测试 session 也会触发一次昂贵的
   consolidation；反之一个通宵 debug 的长 session 永远过不了门。
2. 单一目标函数：buildConsolidationPrompt 一次性灌入所有 session，让一个 forked agent 做"读 + 抽 + 合 + 写"。这是 monolithic dream，token 成本 O(N
   sessions × N memories)，且失败一次全回滚（rollbackConsolidationLock）。
3. 只写不淘汰：MEMORY.md 条目无腐败(decay)、无冲突检测、无"已验证"/"已推翻"标记。上面 CLAUDE.md 自己都说"Memory records can become stale"，但 dream
   本身没有 TTL/confidence 字段。
4. 无反馈回路：dream 产生的记忆事后是否被 memory_recall 命中？命中后是否真的改变了 agent 行为？logEvent('tengu_auto_dream_completed') 只记
   token，不记收益。
5. Fork agent 权限固定只读（extra 里写死"Bash read-only"），但真正的巩固常需写 MEMORY.md 和子文件 —— 依赖 createAutoMemCanUseTool
   做白名单过滤。这种双层约束隐式，未来新增工具很容易漏审。

底层方法论 —— 类比人类睡眠

人类睡眠分 NREM (快波巩固 / replay) + REM (语义抽取 / 图谱重建)。真正重要的不是"睡多久"而是"白天遇到了多少新颖、冲突、情绪强度高的事件"。dream
应被建模成事件驱动的多阶段 pipeline，而不是定时 cron。

升级架构：Dream Pipeline（五阶段 + 证据总账）

┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌───────────┐  ┌──────────┐
│ 1. Capture   │→ │ 2. Triage    │→ │ 3. Replay    │→ │ 4. Weave  │→ │ 5. Audit │
│ 事件落盘     │  │ 新颖/冲突打分│  │ 分片抽取     │  │ 融合&去冲突│  │ 验证&衰减│
└──────────────┘  └──────────────┘  └──────────────┘  └───────────┘  └──────────┘

阶段 1 · Capture（在线，零成本）
- 每个 session 结束时在 transcript header 写一段 DreamEvidence：{ sessionId, novelty, conflicts, userCorrections, surprise, toolErrorRate,
  filesTouched, durationMs }。
- novelty/conflict 由 hooks 增量计算而非事后扫：复用已有 postSamplingHooks + memoryPostToolHook，在 extractMemories 已有的信号管线里打 tag。
- 存储：~/.claude/dream/journal.ndjson（append-only），绝不动 transcript。

阶段 2 · Triage（离线，廉价）
- Dream 门控不再用 minSessions，改用 evidence score = Σ(novelty × 0.4 + conflict × 0.3 + correction × 0.2 + surprise × 0.1)。
- 三档触发：
    - low (<5) → 跳过
    - mid (5–15) → micro-dream：只对最高分 3 个 session 做 Weave，单次 ≤ 5k tokens
    - high (>15) → full-dream：现有 consolidation 路径
- 复用 SideQueryScheduler 的 P3_background + 预算守卫，把 triage 做成后台 task；预算告急自动退 micro。

阶段 3 · Replay（可并行、可恢复）
- 当前是一次性大 prompt；重构为分片 replay：每个 session 产一个 CandidateMemory[]，再由 reducer 合并。
- 分片落盘到 ~/.claude/dream/staging/<dreamId>/，失败可从断点续跑（解决现在"一次全回滚"的脆弱性）。
- 每个 Candidate 带 { type, content, evidence:[sessionId:line], confidence∈[0,1], ttlDays }。

阶段 4 · Weave（冲突图谱）
- 在 reducer 阶段做对称三路比对：new vs existing MEMORY.md vs conflicting candidate。产出下列动作之一：
    - ADD / UPDATE（带 diff & 证据）
    - SUPERSEDE（旧记忆打标 deprecated_by:<id>，不删，保留审计链）
    - CONFLICT_OPEN（写入 ~/.claude/dream/conflicts.md，下次用户对话时作为 question 触达 AskUserQuestion）
- Frontmatter 扩展：
  ---
name: ...
type: feedback
confidence: 0.82        # dream 打分
evidence: [session-abc#L42, session-def#L107]
created_at: 2026-04-09
last_verified_at: 2026-04-09
decay_halflife_days: 30
  ---
- Dream 再也不"删老的写新的"，而是证据总账增量，可回滚到任一历史快照（类似 git reflog）。

阶段 5 · Audit & Decay（被动 + 主动）
- 被动：startRelevantMemoryPrefetch 命中某条记忆时，在 sideQuery 里顺手 recordHit(memoryId)，同时刷新 last_verified_at。
- 主动：下次 dream 的 Audit 阶段对 now - last_verified_at > halflife 的记忆，用 1 次小模型 Q&A 验证其在当前代码下是否仍成立；失败的 confidence *=
  0.7，掉到阈值以下自动 deprecate。
- 这让 memory 系统从"只进不出"变成有呼吸的闭环，根治 stale 问题。

接口契约（对外）

二、Skills 召回机制升级：从"关键词打分 + RRF"→"分层检索 + 自适应工作流卡片"

现状（skillSearch/localSearch.ts + prefetch.ts）

- 单跳检索：scoreSkill 用 name/description/whenToUse 的字面包含计分，叠同义词扩展和 contextScoring（文件扩展名、近期工具），走 RRF 融合。
- 阈值盲区：top-K=5 硬切，分数没有绝对意义。query=空直接返回 0，长自然语言 query 匹配效果骤降（字面命中稀疏）。
- 一次召回即决定：prefetch 只出一次结果存到 attachment，后续轮不会基于 tool trajectory 重召回。
- 没有工作流感知：一个任务常由多个 skill 串联（例如 memory-audit → skillify → commit），但现有打分是独立的 skill 级，没有 "workflow bundle"。
- 冷索引：cachedLocalIndex 是进程级全量 scan，skill 目录很大时冷启动慢（与 MCP lazyLoad 的痛点完全同构，应统一解决）。

底层方法论 —— 三类检索问题叠加

Skill 召回实际上是三个不同的问题被合并了：
1. 明确指令（用户说 /commit 或提到 "commit my changes"）→ 精确/前缀匹配就够。
2. 情境推断（用户说 "clean this up"）→ 需要语义 + 上下文（当前文件、最近 tool 轨迹、过去对这个 repo 用过的 skill）。
3. 工作流串联（完成一个任务需要多个 skill）→ 需要图检索（skill → skill 的共现 / 前驱后继）。

当前代码把三者塞进同一个 scoreSkill，所以任何一类都做得不够。

升级架构：分层召回 + 工作流卡片 + 持续再排序

Query  ──┬─→  Layer-A  Intent Router     （规则 / 结构化）
├─→  Layer-B  Lexical + Context  （现有 RRF，保留）
└─→  Layer-C  Semantic Recall    （embedding / KNN）
│
▼
Fusion (weighted RRF by query class)
│
▼
Workflow Graph Expansion  （一次召一串）
│
▼
Calibrated Threshold Gate
│
▼
Attachment (skills + workflow + confidence)
│
▼
Re-rank on Tool Trajectory  （每 3 个 tool_use 重算一次）

Layer-A · Intent Router（零成本）
- 正则/前缀识别：/<cmd>、"commit / push / merge / review / test"、文件类型关键词（"pdf"、"excel"）。命中即走精确通道，跳过后续昂贵步骤。
- 这一层应在 createSkillSearchSignal 里就打好 signal.intentClass: 'command' | 'inferred' | 'ambiguous'。

Layer-B · 现有 Lexical + Context
- 保持不变；但其权重由 intentClass 动态调节：
    - command → w_lex=1, w_sem=0
    - ambiguous → w_lex=0.4, w_sem=0.6
- 解决现有硬 RRF 导致"字面命中压过语义命中"的问题。

Layer-C · Semantic Recall（延迟启用）
- 每个 skill 的 description + whenToUse 预先嵌入到 ~/.claude/skills-index.vec（启动时异步建索引，复用 SideQueryScheduler P3）。
- 运行期：纯本地 KNN（余弦，top-20）—— 不必接云端 embedding，Xenova/transformers、fastembed 或直接用 hash-bucket LSH 都可作为 v1。
- 与现有 skillChangeDetector 结合：skill 文件 mtime 变化触发增量 re-embed。

Workflow Graph Expansion
- 新增"skill 共现图"：每次一个 session 里实际用过 skill A → B，在 ~/.claude/skills-cooccur.json 记 edge(A,B) += 1。
- 召回阶段：对 top-K 候选做一步图扩展，把常见后继 skill 作为 workflow card 一起曝光，而不是单 skill：
  Suggested workflow: memoryAudit → skillify → commit
- 这直接复用 services/skillSearch/skillWorkflows.ts（文件已存在，目前只有占位），把它接成真实的共现图驱动。

Calibrated Threshold Gate
- RRF 分数没有绝对意义，因此用校准函数 sigmoid((s - μ)/σ) 把分数映射到概率，低于 τ=0.5 的一律不出，避免塞垃圾 attachment 占 context。
- μ/σ 用最近 500 次 prefetch 的分布在线估计（存 skillSearch/telemetry.ts）。

Trajectory Re-rank
- 当前 prefetch 只在"每次 user 输入"触发一次。升级：每 3 个 tool_use 用一次廉价再评分（不重算索引，只对已召回候选重算 contextScore + 新引入的
  "近似完成度" 信号），然后更新 attachment —— 配合 P1-1 orchestrator 的 post_tool 信号，几乎零成本。
- 这解决了长任务中途需要切换 skill 但 prefetch 永远只看首轮 query 的问题。

}

复用/灰度

- 复用：现有 tokenizer、synonyms、contextScoring、workflowTracker、SideQueryScheduler（index 预热 & re-rank 均走 P2/P3 通道）。
- 开关：CLAUDE_SKILL_RECALL_V2, CLAUDE_SKILL_SEMANTIC=1, CLAUDE_SKILL_WORKFLOW_CARDS=1，每层独立，默认 OFF。
- 影子模式：两条 pipeline 并行跑，对比 top-5 重叠率、用户采纳率（是否真的 Skill.invoke）、误召率（召而未用）。
- 度量：skill.recall_precision@5, skill.semantic_contribution, skill.workflow_cards_used, skill.trajectory_rerank_swap_rate。

三、自动运行本地代码机制升级：从"单步 Tool Call"→"计划-执行-验证 的 Harness"

现状（BashTool + permissions + QueryEngine）

1. 单步原子：LLM 每次产出一个 Bash tool_use，harness 执行一条命令、回灌 stdout/stderr，继续下一轮。没有"这是计划的第 3 步"的结构感知。
2. 权限三态：allow/deny/ask，bashPermissions.ts 做 prefix/glob 匹配。可靠但僵硬：用户常被一连串 ask 打断，或为了顺畅勾上
   --dangerously-skip-permissions。
3. 沙箱可选：shouldUseSandbox.ts 有 sandbox-exec 逻辑，但仅对判定为"只读"的命令启用。写操作仍直接落真实 FS。
4. 错误回环薄：命令失败 → 把 stderr 塞回模型 → 模型自己决定要不要重试。没有结构化的失败分类（transient vs 语法 vs 权限 vs 环境缺依赖）。
5. 无验证闭环：改完代码不自动跑 typecheck / test / restore-check。bun run dev:restore-check 存在但要用户手动触发。
6. blast radius 不可视：用户看到 rm -rf build/ 时没有"这条命令会删除 127 个文件、其中 3 个未 commit"的预览。

底层方法论 —— harness 工程的三层分离

一个真正稳健的执行 harness 必须显式分开三层，否则 LLM 永远在"意图/动作/后果"间打结：

┌──────────────────────────────────────────────────┐
│ Intent Layer     "我想让测试通过"                 │
├──────────────────────────────────────────────────┤
│ Plan Layer       Step1:改foo  Step2:跑test  ...  │  ← DAG, 可 replay
├──────────────────────────────────────────────────┤
│ Action Layer     bash/edit/read — 带 pre/post    │
│                  contract + blast radius + dry   │
└──────────────────────────────────────────────────┘

当前代码把 Plan 层完全丢给 LLM 的隐式推理，harness 只看到 Action。升级的核心就是让 Plan 显式化并由 harness 持有。

升级架构：Plan-Execute-Verify Harness（PEV Harness）

组件一：Action Contract（所有工具的统一前置/后置）
interface ActionContract {
dryRunPreview(): Promise<BlastRadius>   // 不执行，只描述影响
preconditions(): Condition[]            // 必须成立才执行
postconditions(): Condition[]           // 执行后必须成立
reversibility: 'reversible' | 'partially' | 'irreversible'
classifyFailure(err): 'transient'|'precondition'|'syntax'|'permission'|'env_missing'
}
- BashTool 实现该接口：dryRunPreview 解析命令→枚举受影响文件/进程/网络端口（复用现有 commandSemantics.ts）。
- 对 rm, mv, git reset --hard 等 irreversible 动作强制弹 BlastRadius UI（展示文件数/行数），而不是只弹命令字符串。

组件二：Plan Graph（显式计划）
- 引入 PlanTool（不调用 LLM，harness 内部结构）：
    - 当用户目标模糊或复杂（判据：task description > 一定 token / 匹配到 "实现/重构/修复" 等意图词） → harness 先让 LLM 吐一个 Plan { nodes: Step[],
      edges: Dep[] }。
    - 每个 Step 带 { id, description, expectedOutputs, verify: Check[] }。
    - Plan 存 ~/.claude/plans/<sessionId>.json，可中断、可 resume、可 diff（类似 terraform plan）。
- 这不是 todo list (TaskCreate)。TaskCreate 面向用户可读进度；Plan 面向 harness 执行调度，带依赖图和回滚点。

组件三：Verify Loop（默认开）
- 每个 Step 的 verify 是可执行谓词：
    - 代码改动 → tsc --noEmit / bun run dev:restore-check / bun test <file>（本项目无 test，但可跑 bun build --no-bundle <touched> 做 transpile check）
    - Bash → postcondition check（test -f, exit code, file hash 变化）
- 失败：按 classifyFailure 分类驱动策略：
    - transient → 自动退避重试（上限 3）
    - env_missing → harness 触发 install-dep 子计划
    - precondition / syntax → 回灌结构化错误给 LLM 重改，不是直接把 stderr 糊过去
    - permission → 升级到用户确认
- 所有 verify 结果写入 Plan Graph 的 runHistory，未来 dream 可以挖出"哪些 verify 反复失败"作为证据。

组件四：Rollback / Snapshot
- Harness 在进入任何 irreversible step 前创建轻量 FS 快照：
    - git repo → git stash -u 或创建 refs/claude/snapshots/<planId>
    - 非 git → 目标文件 copy 到 ~/.claude/snapshots/<planId>/
- Plan 失败或用户 Ctrl+C → 提供一键 rollback planId，对应到 ExitWorktree / git reset --hard <snapshot>。

组件五：Adaptive Sandboxing
- 现状只对只读命令开沙箱。升级为三档沙箱：
  a. readonly-sandbox：只读命令，sandbox-exec 禁写（现状）
  b. scratch-sandbox：写操作只落一个 overlay 目录（macOS 用 firehose / Linux 用 overlayfs，退化方案：先 copy 到 temp 跑再 diff 回来），产出 diff
  预览给用户
  c. direct：用户对该 step 显式 approve 后直跑
- LLM 不感知沙箱档位，由 harness 基于 BlastRadius + 用户历史偏好自动选档。

组件六：Budget & Circuit Breaker
- Plan 级预算：maxWallClockMs、maxTokens、maxToolCalls。超限触发 compact-or-abort（复用 P1-1 Orchestrator）。
- 单命令连续失败计数 → 熔断（复用 sideQuery/circuitBreaker.ts），防止模型陷入"同一条 npm install 重试 12 次"的死循环。

组件七：Observability Shell
- 每个 action 打一条结构化 event：action.started/ended + contract.precond_met + blast_radius + failure_class。
- /doctor 面板显示：当前 plan graph + 最近 10 次 action contract 违反 + 熔断命令列表。

类比现有 harness 的差异

- Claude Code 现状：事实上是 "tool call loop + 弱权限系统"。
- Cursor Composer / Aider：有隐式 plan + diff 预览，但 verify 层薄。
- OpenHands (a.k.a. OpenDevin)：强 plan 层但沙箱不够灵活。
- 本方案：显式 Plan Graph + 合同化 Action + 分档沙箱 + 证据化 Verify —— 与 P1-1 Orchestrator、P0-1 SideQuery、P1-2 MCP lazyLoad
  同构（决策/调度/降级三件套），可以共用同一套 feature flag 与 telemetry 基础设施。

接口契约

interface PEVHarness {
buildPlan(goal: string, ctx: PlanCtx): Promise<PlanGraph>
executeStep(step: Step): Promise<StepOutcome>
verifyStep(step: Step, outcome: StepOutcome): Promise<VerifyReport>
rollbackTo(snapshotId: string): Promise<void>
budget: BudgetGuard            // 复用 sideQuery/budget.ts
breaker: CircuitBreaker        // 复用 sideQuery/circuitBreaker.ts
}
interface StepOutcome {
exitCode: number; stdout; stderr
blastRadius: BlastRadius
failureClass?: FailureClass
artifacts: ArtifactRef[]       // 改动的文件路径 + 行范围
}

复用/灰度

- 复用：commandSemantics / destructiveCommandWarning / shouldUseSandbox / bashPermissions / sideQuery/circuitBreaker / sideQuery/budget，以及 P1-1
  orchestrator 作为预算决策源。
- 开关阶梯：
  a. CLAUDE_PEV_DRYRUN=1：只开 Action Contract 的 dry-run 预览（用户看到 blast radius，但执行路径不变）
  b. CLAUDE_PEV_VERIFY=1：加 verify loop（失败自动分类回灌）
  c. CLAUDE_PEV_PLAN=1：显式 plan graph（影子模式 = 只记录 plan，不用其调度）
  d. CLAUDE_PEV_SNAPSHOT=1：快照 + rollback
  e. CLAUDE_PEV_SANDBOX_OVERLAY=1：scratch-sandbox
    - 每一档都独立可回退，任何一档异常都退回到现状。
- 度量：pev.plan_nodes, pev.verify_pass_rate, pev.rollback_count, pev.precondition_violation, pev.sandbox_downgrade, pev.failure_class_distribution。

四、举一反三：三条链路背后的同一套基础设施

这三个机制看起来风马牛不相及，重构后其实都落在同一套"决策-调度-降级-证据"四元骨架上：



┌─────────────────┬───────────────────────────────────┬────────────────────────────────────────────────┬────────────────────────────────┐
│      原语       │          Dream Pipeline           │                Skill Recall V2                 │          PEV Harness           │
├─────────────────┼───────────────────────────────────┼────────────────────────────────────────────────┼────────────────────────────────┤
│ 决策 (Decide)   │ TriageDecision（micro/full/skip） │ IntentClass + Fusion weight                    │ PlanGraph                      │
├─────────────────┼───────────────────────────────────┼────────────────────────────────────────────────┼────────────────────────────────┤
│ 调度 (Schedule) │ SideQuery P3                      │ SideQuery P2 (prefetch) / P3 (re-embed)        │ PEV runtime + budget           │
├─────────────────┼───────────────────────────────────┼────────────────────────────────────────────────┼────────────────────────────────┤
│ 降级 (Fallback) │ full→micro→skip                   │ semantic→lexical→command                       │ direct→scratch→readonly→reject │
├─────────────────┼───────────────────────────────────┼────────────────────────────────────────────────┼────────────────────────────────┤
│ 证据 (Evidence) │ DreamJournal + conflict graph     │ recall telemetry + cooccur graph               │ runHistory + snapshots         │
├─────────────────┼───────────────────────────────────┼────────────────────────────────────────────────┼────────────────────────────────┤
│ 熔断 (Breaker)  │ lock + rollback                   │ 无（目前），建议新增：召而不用率高→停 semantic │ CircuitBreaker（已复用）       │
└─────────────────┴───────────────────────────────────┴────────────────────────────────────────────────┴────────────────────────────────┘

启示：
1. sideQuery/circuitBreaker.ts 和 sideQuery/budget.ts 应该被提到 services/harness/primitives/，三大机制共享同一套实例；/doctor 一张面板统一观测。
2. "证据账本"应该统一成一个 ~/.claude/evidence/*.ndjson 存储层，Dream 用、Skill 用、PEV 用 —— 以后 dream 里挖出"哪类 skill 最常与哪类 verify
   失败共现"都是免费的。
3. "三档降级 + 影子模式"应该成为本仓库新增机制的统一规范：先影子、再灰度档位、最后切流，和 P0-1/P0-2/P1-1/P1-2 已经建立的节奏保持一致。
4. LLM 的隐式决策能力越强，harness 越应该把"决策点"显式化、账本化 —— 因为影子模式需要可对比，证据账本需要可回放，灰度切流需要可回退。这是 harness
   工程区别于 prompt 工程的根本分界。
