# CJK 正则 `\b` 陷阱与变通

## 适用场景

写涉及**中文(或日文/韩文)文本**的正则时,下面任一模式都可能踩坑:

- `\b(什么|如何|哪个)\b` —— 把英文"word boundary"习惯直接套到 CJK
- `(\w+)(吗|呢)` —— 想用 `\w` 匹配"一个中文词",结果根本匹配不到
- `\B(是|为)` / `\b是\b` —— 想在 CJK 文本里界定"词边界"
- 测试中 `'什么是闭包'.match(/\b什么\b/)` 返回 null,调试一个小时找不到原因

## 核心原因:JS 正则里 CJK 不是 `\w`

`\b` 的定义是:**`\w`(字母/数字/下划线) 与 `\W`(其它) 的交界**。

| 字符 | 在 JS `\w` 里? |
|---|---|
| `a` `Z` `0` `_` | ✅ |
| 空格 `,` `.` `?` | ❌ |
| `什` `吗` `漢` (CJK) | ❌ |

所以:
- `\ba\b` 匹配 " a " 里的 `a` —— 空格是 `\W`,`a` 是 `\w`,有边界 ✓
- `\b什\b` 匹配 `"什"` 附近的"边界" —— 两侧都是 `\W`(CJK 本身是 `\W`),**没有 `\w`↔`\W` 过渡,`\b` 永不匹配** ✗

直白说:**CJK 之间 `\b` 不存在**,`\b什` 只能匹配 `x什` / `空格什` / `串首什` 等紧邻 ASCII 字母数字的情况,而不能匹配自然的"中文字符串开头是'什么'"。

## 真实案例(2026-04, M4 修复)

`accountability-contract.ts` 原来的疑问词识别:

```ts
// ❌ Bug
const QUESTION_INDICATORS = /\b(什么|怎么|如何|哪个|吗|呢)\b/
// 测试:
//   "什么是闭包" → null (😱)
//   "都完成了吗" → null (😱)
//   "What is X?"  → "What"(命中 \b w) ✓
```

"什么是闭包" 失败原因:`\b什` 要求 `什` 左侧是 `\w`。但串首的 `什` 左侧是**字符串起始**,起始前后 `\W`/`\W` 都算,没有过渡,`\b` 不成立。

**修法**(3 段式正则重构,CJK 不带 `\b`,只英文段留 `\b`):

```ts
// ✅ Fixed (src/.claude/hooks/accountability-contract.ts:35-43)
const QUESTION_INDICATORS = new RegExp(
  [
    '[??]',                                    // 1. 任意位置的问号(半/全角)
    '(什么|怎么|如何|哪个|哪些|谁|是否|能否)',  // 2. CJK 疑问词 —— 不带 \b
    '(吗|呢|嘛)[\\s!。.!?]*$',                 // 3. 句尾语气词 —— 用 $ 锚点代替 \b
    '^\\s*(what|how|why|which|is|are|can)\\b', // 4. 英文开头疑问词 —— 可用 \b
  ].join('|'),
  'i',
)
```

- CJK 段**去掉 `\b`**:反正 CJK 左右都是 `\W`,去掉 `\b` 不会多误匹配
- 需要"词尾"语义时,用 `$`(行尾)或 `[\s标点]` 显式列出后置字符类代替 `\b`
- 英文段继续用 `\b`:这是 `\b` 的原生应用场景

## 通用修法速查

| 想表达 | ❌ CJK 下不能用 | ✅ 替代方案 |
|---|---|---|
| "什么"作为单词出现 | `\b什么\b` | 直接 `什么` —— CJK 间不存在 "词" 概念,子串匹配已足够 |
| "是"后跟非 CJK 字符 | `是\b` | `是(?![\u4e00-\u9fff])` 或 `是[\s,。$]` |
| 句尾 "吗" | `吗\b` | `吗$` / `吗[\s!。.!?]*$` |
| 句首 "请" | `\b请` | `^请` / `(^|[\s,])请` |
| "某中文词"(边界受控) | `\w+` | `[\u4e00-\u9fff]+` 或 `[^\s\p{P}]+` |

## 设计清单(写含 CJK 的正则前过一遍)

1. **每个 `\b` 左右都是什么字符?** 左边是英文字母 → 用 `\b` 保留;左边是 CJK 或串首 → 去掉 `\b`。
2. **能不能用显式锚点替代?** `^`(行首) / `$`(行尾) / `[\s,。]`(显式分隔类) 在 CJK 下行为稳定,不依赖 `\w` 定义。
3. **需要 Unicode 属性吗?** 现代 JS 支持 `/\p{Script=Han}/u`、`/\p{P}/u`(标点)。一旦涉及复杂 CJK 分词,优先用这些代替手写 `[\u4e00-\u9fff]`。
4. **有没有混合 CJK 和 ASCII 的输入?** 比如 "请 fix login" —— `\b` 在 ASCII 段可用,CJK 段不可用。分段拼接是最稳的做法。
5. **测试用例覆盖了"串首/串尾/单独"吗?** `'吗'`、`'吗?'`、`'完成了吗'`、`'完成了吗。'` 四种形态都要过测,避免锚点错位。

## 调试一行脚本

```bash
bun -e "
const re = /你的正则/
for (const p of ['串首场景', 'X串中Y', '串尾场景 ','  带空格','串首X','X串尾']) {
  console.log(JSON.stringify(p), '→', re.test(p), '|| match:', JSON.stringify(p.match(re)))
}
"
```

遇到 `null` 时,把对应样本左右补一个 ASCII 空格再测一次,若加空格就通过,**100% 是 `\b` 在 CJK 边不成立**的问题。

## 触类旁通

同类陷阱还有:

| 误用 | 后果 | 正确做法 |
|---|---|---|
| `[a-zA-Z0-9]+` 想抓"中英混排单词" | 漏掉所有 CJK | `[\p{L}\p{N}]+/u` 或 `[^\s\p{P}]+/u` |
| `\s` 想匹配"中文全角空格" | 可能漏掉 U+3000 | 用 `[\s\u3000]` 或依赖 ES2018+ 的 `\s` 已含 U+3000(但 node 行为仍需测) |
| `\d` 想匹配"中文数字 一二三" | 漏掉 | 显式 `[0-9一二三四五六七八九十零]` |
| `\W+` 切分"CJK 词" | 每个字都被切开 | 用分词库(jieba 等)或 Intl.Segmenter |
