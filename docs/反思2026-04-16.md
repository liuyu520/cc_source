四、从“上帝视角”看，真正该收口成什么模型

我建议你脑中把整个系统抽象成一个统一公式：

统一公式

Session Runtime Contract = Provider Identity + Auth Mode + Model Resolution + Prompt Policy

也就是一个会话最终应该由 4 个明确字段定义：

type RuntimeContract = {
provider: 'firstParty' | 'thirdParty' | 'codex' | ...
authMode: 'api_key' | 'oauth' | 'proxy_oauth' | ...
resolvedModel: string
promptPolicy: 'full' | 'third_party_minimal' | ...
}

然后所有地方都不应该自己猜，而应该：

- 统一构造这个 contract
- 统一消费这个 contract

  ---
为什么这是底层最优解

因为你现在的所有问题，本质上都是这些字段没显式化：

例子 1：Codex 为什么像 third-party？

因为现在系统没有把：

- provider=codex
- promptPolicy=full

当成一个显式 contract

而是让不同代码路径自己去推断。

  ---
例子 2：OAuth 代理为什么认证像 OAuth，但 prompt 像 third-party？

因为系统实际上是：

- provider=thirdParty
- authMode=proxy_oauth
- promptPolicy=third_party_minimal

这在语义上完全合法，但对人类来说非常反直觉。

如果这 3 个字段被显式打印出来，问题瞬间就不困惑了。

  ---
例子 3：模型展示值和真实请求值为什么漂移？

因为系统没有统一的：

- resolvedModel

而是多个地方各自重新算一遍。

五、所以，从方法论上讲，这次修复的“下一层正确动作”是什么

不是继续到处补注释。

而是：

方向 A：抽一个统一的 Codex runtime resolver

至少最小化先做这一层：

resolveCodexRuntime(): {

然后：

- 如有调试输出也复用它

这样会立刻带来 3 个收益

1. 模型解析不再漂移
2. OAuth / API Key 模式不再各处重复推导
3. 未来改默认模型只改一处

这比你现在仅仅加 getCodexModelDescription() 更彻底。

你现在那一步是对的，但还属于局部止血，不是结构收敛。

方向 B：抽一个更高层的 runtime contract

如果你想做更彻底一点：

resolveRuntimeContract(): {
provider: APIProvider

}

然后：

- provider 逻辑不再单独散落
- prompt route 不再靠“再推一次”
- 文档/debug 输出可以直接打印 contract

这会把这次所有认知混乱一锅端掉。
