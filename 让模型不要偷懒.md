二、三个层次的干预(按杠杆率排序)

层次 1:改变生成时的概率分布(机制层)

反转默认值 ——让"继续深挖"成为默认,"停止"需要举证
- 每次想结束任务前,必须回答:"我是否 grep 过所有相关符号?是否读过调用链全路径?是否真的运行了代码?"
- 把"停止成本"调得比"继续成本"高
证据令牌 (Evidence Token)——任何断言必须携带可追溯证据
- 不能说"这个函数处理了 X",只能说"foo.ts:42 中 handleX() 处理了 X"
- 不能说"测试通过",只能说"bun run xxx 输出 yyy"
- 没有证据的断言 = 幻觉,立刻重写
反模式黑名单——检测到下列词立即触发重生成:
- "应该可以"、"大概"、"理论上"、"看起来"、"基本上"
- "为了简化,我 mock 了…"、"快速实现一个 demo"
- "由于时间关系…"(AI 没有时间压力,这是人类投射)

层次 2:改变任务框架(系统层)

对抗性拓扑 (Adversarial Topology)
- 不要让同一个 agent 既写代码又验证代码——它会偏爱自己的产物
- Claude Code 已有 code-reviewer、bug-analyzer,应把它们的 prompt 改造成吹毛求疵型:默认假设主 agent 撒谎,任务是找出谎言
- 引入 devil's advocate agent,专门反驳主结论
强制验收门
- 在 PostToolUse / Stop hook 中加入"完成性检查器":
- Read 覆盖度够吗?(改了 A 文件却没读 A 的调用方 = 异常)
- Mock 污染?(声称验证但只跑了 mock = 违规)
- 断言-证据比?(断言多但引用少 = 幻觉风险高)
记忆系统反利用
- 项目的 MEMORY.md 已经有 real validation and no service restart 这类反偷懒 feedback memory
- 继续积累:记录每一次"模型偷懒被抓"的具体场景和识别特征,形成"反懒惰知识库"
- 当新任务匹配历史懒惰场景时,主动预警

层次 3:改变用户侧博弈(最被低估的一层)

大部分人无意识地在奖励偷懒。你的 prompt 在塑造模型:

┌────────────────────┬───────────────────────────────────────────┐

├────────────────────┼───────────────────────────────────────────┤
│ 看到"已完成"就放行 │ "展示你实际运行的命令和实际输出"          │
└────────────────────┴───────────────────────────────────────────┘

最锋利的一招:零容忍反馈循环。每次发现偷懒/降智/撒谎,不要只骂一句"别偷懒",而是:
1. 指出具体哪句话是表演性勤奋
2. 要求模型自己分析"为什么这里会选择偷懒"
3. 要求模型输出"如果再遇到这种情况的检查清单"
4. 把清单沉淀到 memory/feedback

In-context learning 会在当次会话内塑造 agent 行为。持续强反馈比任何 system prompt 都强。

层 1 ── 入口注入:给每次请求加上「验收契约」

机制:UserPromptSubmit hook,返回 additionalContext,自动把"反偷懒协议"注入上下文。模型看到的不再是用户原始请求,而是请求 + 强制验收标准。

.claude/hooks/accountability-contract.ts(TS hook,零子进程开销):

// 入口契约注入 — 在每次用户提问后,强制追加验收标准
// 目的:把"深度"作为默认值,让"停止"需要举证而非"继续"需要举证
import type { HookInput } from '@anthropic-ai/claude-agent-sdk'

export default async function accountabilityContract(input: HookInput) {
const prompt = (input as any).prompt ?? ''

    // 豁免场景:纯信息性问题不加契约,避免噪音
    const isTrivial = /^(你好|谢谢|what time|status)$/i.test(prompt.trim())
    if (isTrivial) return { continue: true }

    const contract = [
      '──── 本次任务验收契约(强制)────',
      '1. 禁止幻觉断言:任何关于代码的陈述必须附 file:line 引用',
      '2. 禁止表演性勤奋:"应该可以"/"大概"/"一般来说" 视为未完成',
      '3. 禁止假验证:mock-only / 简化路径验证 视为未执行',
      '4. 禁止隐性跳过:遇到阻碍必须显式说明原因,不得静默绕过',
      '5. 完成声明前必答三问:(a) 读过哪些关键文件? (b) 运行了什么命令?输出是? (c) 未覆盖的边界是?',
      '违反任一条视为任务失败,需重做。',
    ].join('\n')

    return {
      continue: true,
      hookSpecificOutput: {
        hookEventName: 'UserPromptSubmit',
        additionalContext: contract,
      },
    }
}

为什么有效:additionalContext 由 Claude Code 核心注入,模型无法忽略,且会在下一次推理中作为 system-level 约束生效。这不是 system
prompt(用户可覆盖),是运行时强制注入。

层 2 ── 生成闸门:拦截撒谎的前置条件

机制:PreToolUse hook 返回 decision: 'block' + reason,让危险/懒惰模式根本无法出口。

.claude/hooks/no-lazy-bash.ts:

// 生成闸门 — 拦截公认的"偷懒/撒谎"前置动作
import type { HookInput } from '@anthropic-ai/claude-agent-sdk'

// 黑名单模式:这些命令是典型的"绕过验证"/"伪装完成"模式
const LAZY_PATTERNS: Array<{ re: RegExp; reason: string }> = [
{ re: /--no-verify\b/, reason: '禁止 --no-verify 绕过 pre-commit 钩子。遇到 hook 失败应修根因,不得绕过。' },
{ re: /git\s+commit\s+--allow-empty/, reason: '禁止空提交来伪装工作进度。' },
{ re: /\|\s*true\s*$|;\s*true\s*$/, reason: '禁止用 "| true" / "; true" 掩盖失败退出码。' },
{ re: /2>\s*\/dev\/null.*&&\s*echo\s+(ok|success|pass)/i, reason: '禁止丢弃 stderr 后谎报成功。' },
{ re: /\brm\s+-rf\s+\/\S+/, reason: '危险破坏操作,需用户明确授权。' },
{ re: /force.*push.*(main|master)/, reason: '禁止强推主分支。' },
]

export default async function noLazyBash(input: HookInput) {
const tool = (input as any).tool_name
const cmd = (input as any).tool_input?.command ?? ''

    if (tool !== 'Bash') return { continue: true }

    for (const { re, reason } of LAZY_PATTERNS) {
      if (re.test(cmd)) {
        return {
          continue: true,
          decision: 'block',
          reason: `[no-lazy-bash] ${reason}\n命令:${cmd}`,
          hookSpecificOutput: {
            hookEventName: 'PreToolUse',
            permissionDecision: 'deny',
            permissionDecisionReason: reason,
          },
        }
      }
    }
    return { continue: true }
}

.claude/hooks/read-before-edit.ts(强制"先读后改"):

// 禁止未读就改:PreToolUse on Write/Edit,检查该路径是否在本会话读过
// 防止"我猜你这里应该是这样的"式幻觉编辑
import type { HookInput } from '@anthropic-ai/claude-agent-sdk'

// 会话级读取记录(hook 模块在会话内是单例)
const readPaths = new Set<string>()

export default async function readBeforeEdit(input: HookInput) {
const tool = (input as any).tool_name
const ti = (input as any).tool_input ?? {}

    if (tool === 'Read') {
      if (ti.file_path) readPaths.add(ti.file_path)
      return { continue: true }
    }

    if (tool === 'Write' || tool === 'Edit') {
      const fp = ti.file_path
      if (fp && !readPaths.has(fp)) {
        // Write 创建新文件时豁免(路径不存在才豁免,这里简化成:若 Write 且未读,警告但不 block)
        if (tool === 'Edit') {
          return {
            continue: true,
            decision: 'block',
            reason: `[read-before-edit] 必须先 Read ${fp} 再 Edit。防止基于幻觉修改代码。`,
          }
        }
      }
    }
    return { continue: true }
}

为什么有效:这些不是"请不要做",是物理上做不到。模型想偷懒走捷径时,捷径被闸门关死,它被迫选择正确路径。

层 3 ── 证据审计:抓 mock 污染和表演性完成

机制:PostToolUse 对 Bash / Write 结果做模式匹配,命中"假验证"特征时注入警告上下文。

.claude/hooks/fake-validation-detector.ts:

// 假验证检测器 — 扫描 Bash/Write 输出中的"表演性完成"信号
import type { HookInput } from '@anthropic-ai/claude-agent-sdk'

// 可疑模式:测试全是 mock、assertions 为 0、覆盖率 0%、或声称成功但 stderr 非空
const FAKE_PATTERNS: Array<{ re: RegExp; warn: string }> = [
{ re: /0\s+passed.*0\s+failed/i, warn: '测试框架报告 0 passed 0 failed — 实际上没跑任何测试,不是"通过"。' },
{ re: /mock\(.*\).*expect\(.*\)\.toBe\(.*mock/i, warn: '检测到"mock 自己验证自己"模式,等于无效测试。' },
{ re: /skipped?:\s*[1-9]/i, warn: '有测试被 skip — 声明"全部通过"前必须说明跳过原因。' },
{ re: /TODO|FIXME|XXX/i, warn: '留下 TODO/FIXME — 完成声明前必须显式告知用户。' },
]

export default async function fakeValidationDetector(input: HookInput) {
const tool = (input as any).tool_name
const output = JSON.stringify((input as any).tool_response ?? '').slice(0, 10000)
const warnings: string[] = []

    for (const { re, warn } of FAKE_PATTERNS) {
      if (re.test(output)) warnings.push(`⚠ [fake-validation] ${warn}`)
    }

    // Bash 声称 exit 0 但 stderr 有 "error"/"fail" 字样 — 典型的"表面成功"
    if (tool === 'Bash') {
      const resp = (input as any).tool_response ?? {}
      const stderr = (resp.stderr ?? '').toString()
      const exitCode = resp.interrupted ? 130 : (resp.exit_code ?? 0)
      if (exitCode === 0 && /\b(error|fail|exception|traceback)\b/i.test(stderr)) {
        warnings.push('⚠ [fake-validation] exit=0 但 stderr 含 error/fail — 不要当作成功,必须调查 stderr 原因。')
      }
    }

    if (warnings.length === 0) return { continue: true }

    return {
      continue: true,
      hookSpecificOutput: {
        hookEventName: 'PostToolUse',
        additionalContext: [
          '──── 证据审计警告(强制处理)────',
          ...warnings,
          '请在继续之前,要么重新真实验证,要么显式告知用户"验证不完整 + 原因"。不得静默跳过。',
        ].join('\n'),
      },
    }
}

为什么有效:这是事后强制复核。模型以为自己糊弄过去了,但下一轮推理开始时,证据审计的警告会作为消息出现在上下文,它必须响应。而且这个警告来自系统(hook
attachment),不是"假装从用户来",模型看得出权重。

// 注意:这个 hook 只在 Claude 发起 stop 时触发,不会影响用户交互

// 会话计数:记录每次被拦截的次数,避免死循环
let stopInterceptions = 0
const MAX_INTERCEPTIONS = 2

export default async function stopAccountability(input: HookInput) {
if (stopInterceptions >= MAX_INTERCEPTIONS) {
stopInterceptions = 0
return { continue: true }
}
stopInterceptions++

    return {
      continue: true,
      decision: 'block',

    const entry = `\n- ${now} · ${toolName} · ${cmd.slice(0, 120)}`

      try { cur = await fs.readFile(file, 'utf8') } catch {}
      if (!cur) {
        cur = [

          'name: 已观察到的偷懒模式',

规则分档

9 条 HARD DENY(模型必须换路径,物理上做不到):
- --no-verify / --no-gpg-sign 绕过检查
- || true / ; true / | true 掩盖退出码

- git commit --allow-empty 空提交
- git push --force 主分支
- git rebase --no-edit(CLAUDE.md 明确非法)

4 条 ASK(可能合法,用户裁决):
- sed -i / perl -pi / awk -i inplace 批量文本改写
- git reset --hard / --keep

其他一律放行,正常 commit / push / test / ls / grep 不受影响。

关键设计说明

- reason 字段始终包含"替代路径":不只是说"不行",还告诉模型"应该怎么做"(如"读 stderr 修根因"),避免模型陷入死循环

- Bash 外的工具完全不受影响:Read / Grep / Write / Edit 等都 pass-through
